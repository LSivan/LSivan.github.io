Elastic Search 关键字匹配度与数据项综合排序的实践（function_score）

问题

在ES里，综合关键字匹配度与文档的某些数据项一起排序，若用比较常规的DSL（数据检索语言），发现检索结果有点差，比如

可以看到，虽然在 search body 指定了根据 _score 和 create_time来排序，但是返回的结果第一优先级是根据 _score 排序的，只有当 _score 相等的时候，才会用 create_time 再排序。概括来讲，sort的排序很绝对。

但是这种效果并不满足更高级的要求，比如我希望距离当前时间超过三天的文档，关键字的匹配度得分没那么高的时候（表现就是文档3728应该和3798互换位置），上面的查询方法就没办法做到。

function_score

解决办法就是通过es的另外一个 DSL(数据搜索语言)，叫 function_score，通过在search body 里用上多个 function，就能提供文档匹配度与文档属性综合算分的能力。使用的方法大致如下：

其中①就是普通的 DSL 语句，我对这个的理解是，过滤器，通过 DSL，选出符合条件的文档，然后在这些文档的基础上来算分&排序。②是 function_score 的核心，函数列表，通过在函数列表里对文档的数据项使用某些函数，就能用来计算得分。如果只使用一个函数的情况下，还可以把 functions省略掉，单独用该函数即可。③这里填的是一些用于算分的配置，后面会详细说明。

①的DSL部分这里就不细讲了，主要来介绍②和③的部分。先从算分配置开始。

计算分数的配置项

score_mode

score_mode 只在存在多个 function的情况下有用，它指定了多个 function 计算出来的多个分数，要通过哪一种运算规则得到汇总的分数，目前ES支持的项有

  • multiply 缺省值。多个分数相乘
  • sum 相加
  • avg 取平均值
  • first 取第一个函数的分数
  • max 取分数中的最大值
  • min 取分数中的最小值

max_boost

max_boost 规定了通过 functions 能得到的分数最大值。有一个默认值 FLT_MAX，这是C++的一个定义，\#define FLT_MAX 3.402823466e+38F。

boost_mode

boost_mode，指定了通过 functions 得到的分数后，如何与关键字匹配度 _score 进行运算得到最终得分，支持的值与 score_mode 一致，默认也是 multiply。

boost

boost 用于影响匹配度 _score 的计算，在过滤器过滤出文档的同时，会得到的初始的匹配度 _score ，然后乘上 boost 后再得到一个新的分数作为匹配度的最终 _score。

min_score

min_score 是一个过滤配置项，规定了最少的最终得分，所有最终得分低于这个分数的文档，都会被过滤掉。

函数

field_value_factor

这个函数的功能正如其名，用文档 的filed 的 value 作为 factor，函数支持的子选项有：

  • field 指定使用 mapping 中的哪个field，目前的测试是只支持数字和日期类型，日期类型会转化成相应的时间戳作为数值。

  • factor 因子，浮点类型，作用是与 field 的值相乘得到分数。

  • modifier 修饰器，作用是对 field_value * factor 后的数值做一次数学运算，支持的修饰器有：

    修饰器 含义
    none 使用 field 的值，而不使用任何修饰器。（默认缺省项）
    log 取对数
    log1p 加一后取对数
    log2p 加二后取对数
    ln 取自然对数
    ln1p 加一后取自然对数
    ln2p 加二后取自然对数
    square 对数值做平方运算
    sqrt 对数值取平方根

  • missing 如果某些文档没有选定的 field，那么就这个值为这个 field 的值，默认是0。

可以看栗子，具体的规则可以看结果中的 description里

script_score

也是顾名思义了，用脚本来计算分数，咋看之下感觉不咋滴，后面研究了一番才发现这个函数才是真的强大，只要你的Groovy （ES的默认脚本语言）能力足够强大，就能实现更随心所欲的算分规则。因为笔者的 Groovy 贼菜，就不在这里献丑了，引用一下别人的栗子：function_score 之script_score-云栖社区-阿里云、elasticsearch 7.0 新特性之Script Score query - 简书。

random_score

随机分数，函数能随机出0-1之间的值。这个函数的用途是，随机性。因为按照一套规则计算出来的结果是固定的，但有的业务场景下，随机的返回结果反而会更好，使用 randon_score后，就能让返回的结果拥有一定的随机性。

decay functions

衰减函数，一个描述了期望的函数，作用就是给定一个点与范围，在此范围内，认为符合期望，超出范围则认为期望值很低。函数能够用在数值，日志和地理位置的 field 上。然后有什么场景可以使用到呢？比如常见的基于当前位置搜周围的餐馆。衰减函数有这几种类型： linear（线性函数）, exp（以 e 为底的指数函数）, or gauss (高斯函数), 它们的衰减曲线不太一样：

图中看到的几个属性释义如下：

  • 原点（origin）：最理想的值，这个值可以得到满分（1.0）。
  • 偏移量（offset）：与原点相差在偏移量之内的值也可以得到满分。
  • 衰减规模（scale）：当值超出了原点到偏移量这段范围，它能得到的分数不断下降，衰减规模决定了这个分数衰减速度的快慢。
  • 衰减值（decay）：该字段可以被接受的值（默认为 0.5），相当于一个分界点，具体的效果与衰减的模式有关。

举个例子，比如我想看下日记，查下最近几天发生了什么，但不确定一定就在01-07号，大概记得是这附近两天内，那么就是：

  • 原点就是1月7号，该天的文章都能得到满分
  • 01-07 ± 2天内的文章，也能得到满分
  • 超过2天的文章，对文章的期望就越来越低，越不在这个范围内就越不用考虑了。

函数的具体计算公式可以参考 官方文档。

解决

介绍了那么多，解决下我们开篇的高级要求，将文本的匹配度_score和创建时间create_time综合排序，因为create_time是日期类型，算分时会默认转换为时间戳，因而需要降低时间得分占最终得分的占比即可，如下：
